Here's a spec outline for a Weekly Warehouse Right-Sizing Alert:

Delivery Mechanism
Snowflake Serverless Alert + Email Notification — a CREATE ALERT that runs weekly (e.g., Sunday midnight) with SCHEDULE = 'USING CRON 0 0 * * 0 UTC'. The alert condition checks if any warehouse exceeds your thresholds. The action sends a formatted report via SYSTEM$SEND_EMAIL or posts to a Snowflake notification integration (Slack, Teams, PagerDuty, etc.).

Report Content (per flagged warehouse)
Section	Detail
Store everything in Use Database Warehouse_Optimizer.Service
Warehouse	warehouse_optimizer_wh, xsmall (new)
Classification	OVERSIZED / UNDERSIZED / SPLIT CANDIDATE
Recommendation	Size down to X / Size up to X / Split into N warehouses
Justification	Key metrics driving the recommendation (see below)
Estimated Impact	Projected credit savings or cost increase
Generated SQL	Ready-to-run ALTER WAREHOUSE or CREATE WAREHOUSE statements
Justification Metrics
Oversized: % of queries completing < 1s, median execution time, credit waste estimate (credits used vs. what a smaller WH would cost)
Undersized: remote spill bytes, queue overload time, p95 execution time, count of queued queries
Split candidate: if a single warehouse serves multiple workload profiles (e.g., short BI queries + long ETL), flag it with distinct user/role/query_type clusters and recommend dedicated warehouses per workload
Key Design Considerations
Baseline trending — compare current week vs. rolling 4-week average so alerts reflect change, not just static thresholds. A warehouse that's always "oversized" at low volume may not warrant action.

Multi-cluster awareness — for multi-cluster warehouses, evaluate cluster utilization (min/max clusters actually used vs. configured) separately from size. Recommend scaling policy changes, not just size changes.

Cost materiality filter — only alert on warehouses above a credit threshold (e.g., >5 credits/week). Noise from tiny warehouses dilutes signal.

Workload segmentation — group queries by QUERY_TAG, ROLE_NAME, or USER_NAME to detect mixed workloads. A "split" recommendation is often more valuable than "size up" when a warehouse serves both fast dashboards and heavy transforms.

Cooldown / acknowledgment — track prior recommendations in a state table so you don't re-alert on the same warehouse every week if no action was taken. Include a "snoozed" or "acknowledged" status.

Data sources: QUERY_HISTORY, WAREHOUSE_METERING_HISTORY, WAREHOUSE_EVENTS_HISTORY (for scaling events on multi-cluster), and WAREHOUSES (SHOW WAREHOUSES for current config).

What I'd Add Beyond Your Current App Logic
p50/p95 execution time — more nuanced than just "< 1s"
Queue wait analysis — not just overload time, but frequency of queuing
Credit-weighted sizing — model what the same queries would cost at one size smaller/larger based on actual runtime distribution, not just the 50% heuristic
Suspend pattern analysis — warehouses that auto-suspend/resume frequently may benefit from different auto-suspend timeouts rather than resizing
