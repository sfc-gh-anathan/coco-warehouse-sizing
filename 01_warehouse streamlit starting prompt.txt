This is a sophisticated Streamlit application built specifically for the Snowflake ecosystem using Snowpark. To reverse-engineer this into a "great prompt," we need to define the technical requirements, the business logic for sizing, and the specific UI/UX components.

---

## The Prompt Spec
**Role:** Senior Data Engineer and Streamlit Developer.
**Objective:** Build a Streamlit application called **"Warehouse Cost Optimizer"** designed to help Snowflake administrators right-size their virtual warehouses based on query performance and utilization patterns from `ACCOUNT_USAGE`.

**Tech Stack:**
* **Frontend:** Streamlit.
* **Data Handling:** Pandas & Snowflake Snowpark (`get_active_session`).
* **Visualization:** Altair (stacked bar charts).
* **Styling:** Custom HTML/CSS for metric "pills."

### 1. Data Logic & Business Rules
The app must classify warehouse performance using the following logic:
* **Oversized Query:** Execution time < 1 second and the warehouse is NOT 'X-SMALL'.
* **Undersized Query:** Remote storage spilling > 0 OR Queued Overload Time > 5 seconds.
* **Right-Sized Query:** Neither of the above.
* **Recommendations:**
    * **SIZE DOWN:** If > 75% of queries are "Oversized" (and current size > X-SMALL). Saves ~50% credits.
    * **SIZE UP:** If "Undersized" queries exceed a user-defined % threshold. Adds ~100% credits.

### 2. UI Components & Filters
* **Sidebar:** `selectbox` for Lookback Period (7-90 days), `number_input` for Min Credit Usage, `slider` for Undersized % Threshold, and `multiselect` for Action filters.
* **Main Layout:**
    * **KPI Pills:** Custom HTML cards showing Total Credits, Size Down Savings, Size Up Cost, and Net Impact.
    * **Interactive Table:** Use `st.data_editor` for warehouse metrics with a "Select" column (checkbox) for row-level analysis.
    * **SQL Generator:** A button to generate `ALTER WAREHOUSE` commands for selected rows in a `st.code` block.

### 3. Visualizations (Altair)
Create three time-series stacked bar charts (hourly buckets) that update based on selection:
1.  **Query Classification:** Count of Oversized, Undersized, and Right-Sized queries.
2.  **Avg Execution Time:** Average execution time in seconds.
3.  **Bytes Spilled:** Comparison of Local vs. Remote spilling (KB).

### 4. Technical Constraints
* **Snowpark:** Use `session.sql()` to query `QUERY_HISTORY` and `WAREHOUSE_METERING_HISTORY`.
* **Performance:** Use `st.cache_data` with a 600s TTL.
* **Logic:** Handle warehouse size ordering (X-SMALL to 6X-LARGE) for sizing calculations.
* **Filtering:** Exclude system warehouses (e.g., `SYSTEM$` or `COMPUTE_SERVICE_WH`).
